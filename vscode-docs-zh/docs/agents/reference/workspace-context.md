# Copilot 如何理解你的工作区

Copilot 代理会搜索你的整个代码库，理解组件之间的关联，并根据你的实际代码提供答案。你可以使用宽泛的提示，例如"在哪里处理身份验证？"或"为列表端点添加测试"，然后获得基于代码库的准确答案和编辑。

本文介绍 Copilot 如何理解你的代码库，包括其用于搜索代码的不同工具以及为实现快速搜索而创建的索引。无论代码库只有 5 个文件还是 500,000 个文件，这些通用方法都会被采用。

## 搜索和读取工具

当你发送提示时，Copilot 会分析所需的信息，并自动选择合适的搜索工具组合来回答你的问题或开始生成代码编辑。Copilot 会为此运行多个工具，查看结果，并自动执行后续搜索，直到对问题有充分的理解。

例如，当要求"为支付服务添加错误处理"时，代理可能会：

1. 使用**语义搜索**在整个项目中查找与支付相关的代码。
1. 使用 **grep** 查找代码库中现有的错误处理模式。
1. 使用**用法**追踪支付函数的调用方式。
1. 使用**文件搜索**定位相关的配置和测试文件。
1. 读取相关文件并进行协调的更改。

这种迭代方法意味着代理以与开发人员相同的方式收集上下文：从多个角度探索代码库，直到获得完整的图景。

代理可以使用以下内置搜索工具。你也可以在提示中显式引用这些工具，方法是输入 `#` 后跟工具名称。

| 工具 | 描述 |
|------|-------------|
| **语义搜索** (`#codebase`) | 查找与你问题含义匹配的代码，而不仅仅是精确的关键词。需要[工作区索引](#语义索引来源)。 |
| **文本搜索** | 在文件内容中搜索文本匹配项，例如特定关键词。 |
| **Grep** | 跨文件搜索精确文本或正则表达式模式。无需索引即可工作。 |
| **文件搜索** | 通过文件名或 glob 模式查找文件。 |
| **用法** | 结合"查找所有引用"、"查找实现"和"转到定义"来追踪符号在文件间的使用方式。 |
| **列出目录** | 列出目录内容以探索项目结构。 |
| **读取文件** | 读取特定文件的内容以详细检查其代码。 |

这些工具适用于任何工作区大小。对于小型项目，整个工作区可以直接读入代理的上下文中。对于较大的项目，代理会根据项目大小和可用资源选择最高效的搜索策略。

### 搜索工作区之外的 GitHub 仓库

除了搜索当前工作区，代理还可以搜索其他 GitHub 仓库中的代码。当代理需要了解本地未打开的代码库时，这很有用，例如查找上游库中的 API 或跨仓库比较实现。

| 工具 | 描述 |
|------|-------------|
| **语义搜索 GitHub 仓库** (`#githubRepo`) | 对 GitHub 仓库进行语义搜索，查找相关的源代码片段。使用 `owner/repo` 指定仓库。 |
| **GitHub 文本搜索** (`#githubTextSearch`) | 对 GitHub 仓库或组织进行文本搜索，查找包含特定关键词或代码模式的文件。 |

对于更高级的 GitHub 工作流，例如搜索和管理议题或拉取请求，请使用 [GitHub MCP 服务器](https://github.com/github/github-mcp-server)。

### 代理可以访问的内容

代理会搜索开发人员在浏览代码库时会使用的相同来源：

* 工作区中所有[可索引文件](#语义索引中包含什么内容)，除了被 `.gitignore` 文件忽略的文件
* 包含嵌套文件夹和文件名的目录结构
* 代码符号和定义（类、函数、变量）
* 活动编辑器中当前选中的文本或可见的文本
* 对话历史记录和之前的工具结果

> [!IMPORTANT]
> 如果你打开了被忽略的文件或在其中选中了文本，则 `.gitignore` 会被绕过。

## 语义搜索

语义搜索工具 (`#codebase`) 通过含义而非精确关键词来查找代码。

语义搜索需要构建和维护索引，以便即使在非常大的代码库上也能快速运行。Copilot 会自动为你维护此索引。在幕后，索引的部分内容可能存储在你的机器上，部分可能来自远程源，但你无需管理这种区别。

你可以在 VS Code 状态栏中的 Copilot 状态仪表板中查看索引状态。

![Screenshot showing the workspace index status in the Copilot status menu.](../images/workspace-context/workspace-index-status.png)

### 语义索引来源

Copilot 为任何工作区自动构建和维护语义索引。索引来源决定了索引首次可用的速度：

* **GitHub 仓库**：GitHub 为你的工作区中的 GitHub 仓库建立索引。使用 GitHub 帐户登录即可使用它们。此索引每个仓库只需构建一次，这意味着它通常立即可用。GitHub 会在需要时构建和更新此索引。对于小型和中型项目，速度很快，但如果你的仓库包含数十万个文件，可能需要一些时间。远程索引适用于托管在 GitHub.com 或 GitHub Enterprise Cloud 上的仓库，不支持 GitHub Enterprise Server。

* **Azure DevOps 仓库**：索引会自动构建和维护。在 VS Code 中使用 Microsoft 帐户登录，Copilot 即可开始使用该索引。检查 Copilot 状态栏项以获取当前索引状态。

* **其他工作区**：对于任何其他工作区，包括不受 GitHub 或 Azure DevOps 仓库支持的本地文件夹，Copilot 会为你构建语义索引。初始构建可能需要几分钟，之后索引会在后台保持更新。此功能目前默认对个人帐户启用，但对组织和企业用户默认关闭。有关如何使用策略启用此功能的更多详细信息，请参阅 [GitHub 文档](https://docs.github.com/en/copilot/concepts/context/repository-indexing#semantic-indexing-for-non-github-repositories)。

要按需为当前工作区构建或重建索引，请从命令面板运行**构建代码库语义索引**命令。

### 语义索引中包含什么内容

VS Code 索引当前项目中相关的文本文件。这不限于特定的文件类型或编程语言。VS Code 会自动跳过一些通常不相关的常见文件类型，例如 `.tmp` 或 `.out` 文件。

工作区索引还会排除通过 `setting(files.exclude)` 设置或 `.gitignore` 文件从 VS Code 中排除的文件。

二进制文件（如图像或 PDF）不会被索引。

## 获得更好结果的提示

你的提示措辞方式会影响代理使用哪些工具以及结果的质量。

* **具体明确**：避免使用模糊的术语，例如"这是做什么的"，其中的"这"可能指上一个回答、当前文件或整个项目。
* **使用代码术语**：使用代码中出现的函数名、类名和概念，以便代理找到精确匹配项。
* **手动添加上下文**：选中代码、引用文件或使用[#-mention 上下文项](/docs/chat/copilot-chat-context.md)，例如调试上下文、终端输出等。
* **限定请求范围**：回答可以来自多个引用，例如"查找没有 catch 块的异常"。但不要期望进行完整的代码库分析，例如"此函数被调用了多少次？"。
* **对非代码问题使用外部工具**：对于"谁对此文件做出了贡献？"等信息，请配置相关的[工具或 MCP 服务器](/docs/chat/chat-tools.md)。

## 私有仓库

要对私有仓库使用语义搜索，Copilot 可能需要额外的权限。如果这些权限尚未授予，VS Code 会在启动时要求授予。一旦授予，会话将被安全存储以供将来使用。

![Modal window asking for additional authentication for a private repository.](../images/workspace-context/authentication.png)

在 [GitHub Copilot 信任中心](https://resources.github.com/copilot-trust-center/)了解更多关于安全性、隐私和透明度的信息。

## 常见问题

### 我需要在提示中使用 `#codebase` 吗？

不需要。代理在合适的时候会自动使用语义搜索。你无需在提示中添加 `#codebase`。

`#codebase` 工具始终是语义化的，并提供一致的结果。如果你想为特定提示强制使用语义搜索，你仍然可以将 `#codebase` 添加为[上下文项](/docs/chat/copilot-chat-context.md)。

### 如果我的工作区没有语义索引会怎样？

大多数工作区会自动建立索引，但如果索引尚不可用（例如，新的本地工作区仍在构建其初始索引），代理仍然可以通过使用文本搜索、grep、文件搜索和语言智能来有效搜索你的代码。工作区索引启用语义搜索，它通过含义而非关键词来查找代码。没有它，代理会依赖其他搜索工具，并且仍然可以为大多数提示提供准确的结果。我们发现这些其他工具仍然能提供很好的结果。
