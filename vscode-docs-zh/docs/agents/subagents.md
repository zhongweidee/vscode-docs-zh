# Visual Studio Code 中的子代理

在处理复杂任务时，你可以将子任务委托给子代理。子代理是一个独立的 AI 代理，执行专注的工作，例如研究某个主题、分析代码或审查更改，并将结果反馈给主代理。

> [!TIP]
> 若要运行生成独立、可分享报告（而非将结果反馈给主对话）的深度研究，请在 Copilot CLI 会话中使用内置的[研究代理](/docs/agents/agent-types/copilot-cli.md#run-deep-research-with-the-research-agent)。

有关子代理概念的背景知识（上下文隔离、同步和并行执行），请参阅[代理概念](/docs/agents/concepts/agents.md#subagents)。

本文介绍了如何在 VS Code 中使用子代理，包括使用场景、调用模式以及如何将自定义代理作为子代理运行。

## 何时使用子代理

以下场景说明了子代理如何改善你的 AI 辅助开发工作流。

<details>
<summary>在实现之前进行研究</summary>

在构建新功能时，使用子代理在研究最佳实践、评估库或分析代码库中现有模式后，主代理再开始实现：

```prompt
执行关于 Node.js 应用程序不同 OAuth 2.0 实现模式的独立研究。
将每种模式与当前实现进行比较，并返回包含优缺点的推荐方案。
```

主代理仅接收最终推荐结果，为其实际的实现工作保持上下文的清晰。

</details>

<details>
<summary>并行代码分析</summary>

在重构或审查代码时，并行运行多个子代理来分析不同方面：

```prompt
分析此代码库，寻找重构机会。并行执行以下任务：
1. 查找重复的代码模式
2. 识别未使用的导出和死代码
3. 审查错误处理的一致性
4. 检查安全漏洞

将发现编译为优先级排序的行动计划。
```

</details>

<details>
<summary>探索多种解决方案</summary>

当你对最佳方法不确定时，使用子代理探索不同选项，避免污染主对话上下文：

```prompt
我需要为此 API 实现缓存。对以下三种方法进行独立研究：
1. 设计一个基于 Redis 的缓存方案
2. 设计一个带 LRU 淘汰策略的内存缓存方案
3. 设计一个分层缓存的混合方案

比较结果并针对我们的用例推荐最佳方法。
```

</details>

<details>
<summary>专项聚焦的代码审查</summary>

使用自定义代理作为子代理，从不同的审查视角进行审视：

```prompt
从不同角度审查此 PR 中的更改。并行执行以下审查：
- 运行 security-reviewer 代理检查漏洞
- 运行 performance-reviewer 代理识别瓶颈
- 运行 accessibility-reviewer 代理验证无障碍合规性

将发现合并为一份统一的审查摘要。
```

</details>

<details>
<summary>多模型共识</summary>

使用不同模型的子代理，获取对同一问题的多元视角：

```prompt
我需要评估支付服务的错误处理情况。
并行运行两个子代理，每个使用不同的模型：
1. 使用 GPT-4o 审查代码的错误处理缺陷
2. 使用 Claude Sonnet 4.6 审查代码的错误处理缺陷

比较两者的发现，标出它们一致和分歧的地方。
```

主代理收集两份结果并综合形成一份组合分析。

</details>

## 调用子代理

### 子代理的调用方式

子代理通常由**代理发起**，而非用户在聊天中直接调用。要允许主代理调用子代理，请确保已启用 `agent/runSubagent` 工具。

默认情况下，子代理自身无法再调用进一步的子代理。要启用递归嵌套，请启用 `setting(chat.subagents.allowInvocationsFromSubagents)` 设置。详见[嵌套子代理](#嵌套子代理)。

主代理自行决定何时需要上下文隔离。你无需为每个任务手动输入"运行一个子代理"。其工作模式如下：

1. 你（或你的自定义代理指令）描述一个复杂任务。
1. 主代理识别出任务中受益于隔离上下文的部分。
1. 主代理启动一个子代理，仅传递相关的子任务。
1. 子代理自主工作并返回摘要。
1. 主代理整合结果并继续工作。

你可以通过提示词来暗示需要子代理委托，如建议进行独立研究或并行分析。主代理将启动一个子代理，将任务传递给它，并仅接收最终结果。

> [!TIP]
> 为获得一致的子代理行为，请在自定义代理的指令中定义何时使用子代理，而非每次都手动提示。

为优化子代理性能，请明确定义任务和预期输出。这有助于子代理聚焦于特定目标，而不会将不必要的上下文传回主代理。

请参阅[使用场景](#何时使用子代理)部分，了解如何构建调用子代理的提示词示例。

### 在提示词文件中调用子代理

要在提示词文件内部调用子代理，请确保在 frontmatter 的 `tools` 属性中包含 `runSubagent` 或 `agent` 工具：

```markdown
---
name: document-feature
tools: ['agent', 'read', 'search', 'edit']
---
运行一个子代理来研究新功能的实现细节，仅返回与用户文档相关的信息。
然后使用新文档更新 docs/ 文件夹。
```

在提示词指令中，你可以通过建议对特定子任务进行独立研究或并行分析来提示代理使用子代理。

### 你在聊天中看到的内容

当子代理运行时，它在聊天中显示为一个可折叠的工具调用。默认情况下，子代理是折叠的，并显示：

* 自定义代理的名称（如果你指定了一个）
* 当前正在运行的工具（例如，"正在读取文件..."或"正在搜索代码库..."）

选择子代理工具调用以展开并查看完整详情，包括子代理执行的所有工具调用、传递给子代理的提示词以及返回的结果。

你可以控制显示多少细节，而不会让中间步骤使主对话显得混乱。

## 将自定义代理作为子代理运行

默认情况下，子代理会继承主聊天会话中的代理，并使用相同的模型和工具。要为子代理定义特定行为，请使用[自定义代理](/docs/agent-customization/custom-agents.md)。自定义代理可以指定自己的模型、工具和指令。当用作子代理时，这些设置将覆盖从主会话继承的默认值。

主代理也可以在调用子代理时请求特定的模型。详情见[为子代理选择模型](#为子代理选择模型)部分。

### 控制自定义代理的调用方式

你可以使用两个 frontmatter 属性来控制自定义代理的调用方式：

* `user-invocable`：控制该代理是否出现在聊天中的代理下拉列表中（默认为 `true`）。设置为 `false` 可创建仅能作为子代理访问的代理。
* `disable-model-invocation`：阻止该代理被其他代理作为子代理调用（默认为 `false`）。设置为 `true` 时，代理只能由用户显式触发。

例如，创建一个只能用作子代理（不显示在下拉列表中）的代理：

```markdown
---
name: internal-helper
user-invocable: false
---

此代理只能作为子代理被调用。
```

> [!NOTE]
> `infer` 属性已弃用。请改用 `user-invocable` 和 `disable-model-invocation` 以实现更精细的控制。

要将自定义代理作为子代理运行，提示 AI 在子代理中使用自定义或内置代理。例如：

* `将 Research 代理作为子代理运行，研究本项目的最佳身份验证方法。`
* `在子代理中使用 Plan 代理，为 myfeature 创建实现计划。然后将计划保存在 plans/myfeature.plan.md 中。`

### 限制代理可以使用的子代理（实验性）

默认情况下，所有未设置 `disable-model-invocation: true` 的自定义代理都可作为子代理使用。如果有两个或更多代理具有相似的名称或描述，AI 可能会选择非预期的代理。

你可以通过在主代理的 frontmatter 中指定 `agents` 属性并提供允许的自定义代理列表，来限制哪些自定义代理可以用作子代理。

`agents` 属性接受：

* 代理名称列表（例如，`['Edit', 'Search']`），仅允许特定的代理
* `*` 允许所有可用代理（默认行为）
* 空数组 `[]` 禁止任何子代理使用

> [!NOTE]
> 在 `agents` 数组中显式列出某个代理会覆盖 `disable-model-invocation: true`。这意味着你可以创建一些代理，使其在一般的子代理使用中受到保护，但仍可供明确允许它们使用的特定协调代理访问。

例如，一个测试驱动开发 (TDD) 代理应仅使用 `Red`、`Green` 和 `Refactor` 代理作为子代理。如果不加以限制，TDD 代理可能会选择一个更通用的编码代理来实现测试，而非使用专门的 TDD 代理。

```markdown
---
name: TDD
tools: ['agent']
agents: ['Red', 'Green', 'Refactor']
---
使用测试驱动开发方法实现以下功能。使用子代理指导以下步骤：
1. 使用 Red 代理编写失败的测试
2. 使用 Green 代理实现代码以使测试通过
3. 使用 Refactor 代理提高代码质量
```

## 为子代理选择模型

当子代理运行时，模型按以下优先级顺序确定：

1. **显式模型参数**：主代理在调用 `runSubagent` 工具时直接指定一个模型。
1. **代理配置的模型**：自定义代理 `.agent.md` frontmatter 中的 [`model`](/docs/agent-customization/custom-agents.md#header-optional) 属性。这可以是一个单一模型名称或一个按优先级排列的模型列表。
1. **主模型**：正在运行父对话的模型。

要为子代理请求特定模型，请在提示词中包含模型偏好：

* `使用 Claude Sonnet 4.6 运行一个子代理，研究此代码库中的身份验证模式。`
* `在子代理中使用 GPT-4o 分析此模块的性能。`

你也可以在自定义代理的指令中定义模型偏好，以一致地将子代理任务路由到特定模型。

> [!NOTE]
> 请求的模型不能超过主模型的费用层级。如果你请求更昂贵的模型，子代理将回退到主模型。

## 嵌套子代理

默认情况下，子代理不能生成进一步的子代理。这可以防止代理在循环中意外调用自身时产生无限递归。然而，某些工作流会受益于递归委托，例如，一个分治代理将一个大任务拆分为更小的部分，并将每个部分委托给自身。

要启用嵌套子代理，请启用 `setting(chat.subagents.allowInvocationsFromSubagents)` 设置（默认为 `false`）。启用后，子代理可以生成自己的子代理，最大嵌套深度为 5。

### 示例：递归代理

递归代理在其自身的 `agents` 属性中列出自身。这实现了分治模式，代理将一个问题拆分为更小的部分，并将每个部分委托给一个新的自身实例。

```markdown
---
name: RecursiveProcessor
tools: ['agent', 'read', 'search']
agents: [RecursiveProcessor]
argument-hint: 要处理的项列表
---

你通过分治方式处理项列表：
- 如果列表中有超过 4 个项目，将其拆分为两半，并将每一半委托给一个 RecursiveProcessor 子代理。
- 如果列表中有 4 个或更少的项目，则直接处理这些项目。
- 将每个子代理的结果合并为最终结果。
```

## 编排模式

子代理支持**编排模式**，协调代理将工作委托给专门的工作代理。这种方法有助于你构建复杂的工作流，同时让每个代理专注于其最擅长的工作。

### 协调器与工作者模式

协调代理管理整体任务，并将子任务委托给专门化的子代理。每个工作代理可以拥有一组定制工具。例如，规划和审查代理只需只读访问权限，而实现者需要编辑能力。

```markdown
---
name: Feature Builder
tools: ['agent', 'edit', 'search', 'read']
agents: ['Planner', 'Plan Architect', 'Implementer', 'Reviewer']
---
你是一个功能开发协调器。对于每个功能请求：

1. 使用 Planner 代理将功能拆分为任务。
2. 使用 Plan Architect 代理根据代码库模式验证计划。
3. 如果架构师识别出可复用的模式或库，则向 Planner 发送反馈以更新计划。
4. 使用 Implementer 代理为每个任务编写代码。
5. 使用 Reviewer 代理检查实现。
6. 如果审查者发现问题，则再次使用 Implementer 代理应用修复。

在规划与架构之间、审查与实现之间进行迭代，直到每个阶段都收敛。
```

工作代理各自定义自己的工具访问权限，并且由于关注范围更窄，可以选择更快或更具成本效益的模型：

```markdown
---
name: Planner
user-invocable: false
tools: ['read', 'search']
---
将功能请求拆分为实现任务。整合来自 Plan Architect 的反馈。
```

```markdown
---
name: Plan Architect
user-invocable: false
tools: ['read', 'search']
---
根据代码库验证计划。识别应复用的现有模式、工具和库。标记任何与现有功能重复的计划步骤。
```

```markdown
---
name: Implementer
user-invocable: false
model: ['Claude Haiku 4.5 (copilot)', 'Gemini 3 Flash (Preview) (copilot)']
---
编写代码以完成分配的任务。
```

此模式使协调器的上下文专注于高层工作流，而每个工作代理具有清晰的上下文和适合其特定工作的适当权限。

### 多视角代码审查

代码审查受益于多角度审视。单次审查往往遗漏问题，而当你从不同视角审视时，问题会变得明显。使用子代理并行运行每个审查视角，然后综合整理发现。

```markdown
---
name: Thorough Reviewer
tools: ['agent', 'read', 'search']
---
你同时从多个视角审查代码。将每个视角作为并行子代理运行，以确保发现独立且无偏见。

当被要求审查代码时，并行运行以下子代理：
- 正确性审查器：逻辑错误、边界情况、类型问题。
- 代码质量审查器：可读性、命名、重复。
- 安全审查器：输入验证、注入风险、数据暴露。
- 架构审查器：代码库模式、设计一致性、结构对齐。

所有子代理完成后，将发现综合为优先级排序的摘要。标注哪些问题至关重要，哪些是锦上添花。肯定代码做得好的地方。
```

此模式之所以有效，是因为每个子代理都以全新的视角审视代码，不会受到其他视角发现的锚定效应影响。在此示例中，编排器通过其提示词塑造每个子代理的关注领域。这是一种轻量级方法，不需要额外的代理文件。

> [!TIP]
> 若需更多控制，每个审查视角可以作为拥有专门工具访问权限的自定义代理。例如，安全审查器可能使用专注于安全的 MCP 服务器，而代码质量审查器可能拥有访问 linting CLI 工具的权限。这种方法让每个视角都能使用最适合其特定关注点的工具。
