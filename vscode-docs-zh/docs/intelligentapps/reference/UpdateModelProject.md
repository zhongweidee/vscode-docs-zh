# 在模型转换中更新模型项目

Foundry Toolkit 中的配方可能会因质量改进、功能更新等原因而更新。更新扩展后，如果 Foundry Toolkit 中的配方版本与本地模型项目中的版本不同，你将看到“需要更新”的通知。

本文介绍了此情况下更新模型项目的不同方法。

![显示模型需要更新的截图](../images/modelconversion/need-update.png)

> [!NOTE]
> Foundry Toolkit 扩展仅包含最新版本的配方以节省空间。扩展的旧版本仍包含旧配方。使用旧版本是处理此问题的方法之一，如下所述。

## 更新模型项目的流程

通常，你有三种选择：

| 方法 | 详情 |
|-|-|
| 自动替换 | 自动将模型项目的内容替换为最新内容。 |
| 迁移更改 | 创建一个新的模型项目，并将旧模型项目中的更改迁移到新项目中，反之亦然。 |
| 降级 Foundry Toolkit | 降级 Foundry Toolkit 扩展以继续使用之前的版本。 |

### 自动替换

当你未手动修改任何内容时，建议使用此方法。单击“更新”，模型项目的内容将自动替换为最新内容。更新过程中不会影响历史记录。

### 迁移更改

当你进行了更改时，建议使用此方法。首先使用相同的模型 ID 创建新模型项目，然后你可以将更改迁移到新项目中，或将新项目的内容迁移到现有项目中。

对于后一种方法，迁移后在 `model_project.config` 中将 `version` 设置为与新项目匹配的值，这样“需要更新”的通知就会消失。后一种方法更复杂，但它允许你在可能时重用 Olive 缓存。

### 降级 Foundry Toolkit

此方法可作为临时变通方案使用，同时你可以在其他选项中进行选择，因为它不允许你使用新配方或新功能。

## 对模型历史记录的影响

更新后，之前版本的历史记录会变灰，因为它们已过时。

![显示历史记录已过时的截图](../images/modelconversion/history-outdated.png)

历史记录中的模型和推理示例仍然有效可用。或者，你可以重新运行工作流来生成新的记录。如果工作流没有太大变化，由于之前的结果已被缓存，运行速度会很快。

显示参数的 UX 可能无法正常工作。在这种情况下，你可以查看历史记录文件夹中的 `olive_config.json` 来获取参数。

## 模型项目版本的含义

当 [olive-recipes](https://github.com/microsoft/olive-recipes) 中的配方更新时，版本号会增加。配方可能在两次 Foundry Toolkit 发布之间更新多次。因此，当你看到版本号时，可能会发现数字不是连续的。

出于同样的原因，更新可能影响一个、全部或为零的工作流——有时只是推理示例更新而没有工作流更改。因此，总体而言，如果更新后现有历史记录中的已转换模型仍然运行良好，我们建议继续使用它。
